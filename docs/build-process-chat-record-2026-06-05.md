# 一个活动预订小程序是怎么做出来的

记录日期：2026-06-05  
记录形式：沟通过程整理版  
用途：给其他人看这个小程序从想法、截图、讨论、试错、修正，到逐渐成型的过程。

## 开始：从“我想做一个自己的小程序”说起

一开始，用户拿着一个正在使用的旅游线路预订小程序来问：如果只有使用权，想照着它的功能开发一个自己的小程序，应该怎么向我说明。

我们先把问题拆开：不是只看几张页面截图，也不是只说“我要一个旅游小程序”，而是要整理三类东西：

1. 功能流程：客户怎么浏览、预约、支付、取消、评价；后台怎么建活动、排班、查订单。
2. 页面截图/录屏：客户界面、后台界面、管理端界面分别是什么样。
3. 字段和规则：哪些字段要保留，哪些不要；退款、排班、库存、权限怎么计算。

用户问录屏有没有帮助，我确认有帮助，尤其是配合语音解释，可以看到真实操作路径、页面跳转和“不顺手”的地方。

这一步很重要，因为我们不是一开始就写代码，而是先让原有系统“暴露结构”。

## 第一轮理解：客户小程序、后台和管理端三套系统

用户先给了客户界面录屏和后台截图。我们逐渐确认这个系统不是一个简单预约页，而是三套端：

- 客户端小程序：看活动、选日期、下单、支付、取消、评价。
- 后台网页：建活动、排班、看订单、管理顾客、装修首页。
- 管理小程序：领队或子账户在手机上处理订单、接收通知。

最早用户说“门店”，后来经过讨论，我们把“门店”改成“组”。因为用户不是经营地理门店，而是按活动类型分组，比如徒步、水上运动、手工、亲子等。

这是第一个关键改造：不是照抄原系统的概念，而是把它改成适合用户自己业务的结构。

## 活动管理：从复杂表单删到真正需要的字段

用户发了新增活动截图。原系统的字段很多：服务分类、支付类型、预约资料表、状态、封面、服务信息等。

我们逐项筛选：

- 保留活动名称。
- 保留活动照片，但希望多图上传、能调整顺序、不限制尺寸。
- 服务分类改成标签，允许多个标签。
- 提前预约时间单独写小时数。
- 不需要每个活动单独预约资料表，统一收集少量客户信息。
- 不需要“停用”状态，因为没有排班或暂停排班就等于不可预约。

用户提出活动介绍可以加常用字段，比如适合年龄、集合地点、海拔上升。我建议这些字段分两类：

- 稳定结构化字段：集合地点、适合年龄、海拔上升、语言、是否含餐。
- 灵活标签：适合新手、亲子、小狗同行、雨天不适合等。

后来我们又加了领队微信、可导航集合地点、领队档案等。

这里的思路是：能被筛选和结构化展示的东西，不要藏在大段文字里；真正讲故事的部分，才放进富文本介绍。

## 排班：从一次性日期改成长期星期规则

排班是整个系统里最复杂的地方之一。

一开始原型里把排班做得偏一次性，用户马上指出：排班不是一次性的，要以星期为标准。比如选择每周二、四、六，那么以后永久就是这个节奏，除非停止排班或用休息日干预。

随后我们不断修正：

- 每周不同日期可以有不同时间。
- 每天可以多个时间段。
- 时间段可以重叠，因为可能不同领队带。
- 不再跨三天，只做一天内 00:00-24:00。
- 时间以 15 分钟为单位，只显示 00、15、30、45。
- 排班不能重复新增，新增一次后就是编辑。
- 整个排班可以暂停，但保留设定。
- 暂停排班和没有排班的活动，在后台排到后面。

休息日和特殊排班也经历了几次修正：

- 新增休息日后，不应该跳回排班页，而要停留在休息页。
- 特殊排班只选一天，不需要日期区间。
- 选择特殊排班日期后，要先把那天原本的排班导进来，再修改。
- 特殊排班优先于休息日。

这一轮最大的收获是：排班不是“日期表”，而是一套长期规则 + 临时例外。

## 订单和退款：把“核销”从流程里拿掉

原系统有核销按钮、二维码、扫码核销等概念。用户明确说不需要。

我们把订单流程简化为：

- 待付款。
- 已预约。
- 已完成。

客户支付成功后直接进入已预约，不需要人工审核。活动结束时间后自动完成。

退款规则也不断收敛：

- 客人只能活动开始前取消。
- 管理员或授权子账户取消，活动开始前全额退款。
- 客人取消时根据规则扣款，原路退回微信。
- 天气原因不要让客人自己硬取消，要提示联系领队，由领队帮忙取消并全额退款。

退款弹窗的文案也改了几轮：

- 需要显示退改规则和扣款金额。
- “款项原路退回”放在“退款比例”后。
- 删除多余说明。
- 原位置改成天气原因说明。

这种细小文案改动看起来小，但对真实用户体验很关键。它决定客人在取消时是否清楚、安心、少误操作。

## 顾客管理：保留冻结，但不告诉客人冻结

顾客管理也做了大量删减。原系统有性别、标签、积分等，用户都不要。

最后保留：

- 顾客基础信息。
- 钱包余额。
- 虚拟充值。
- 冻结功能。

有一个很现实的业务规则：冻结后客人不能下单，但前台提示“系统故障”，不要提示“你被冻结了”。原因是用户希望筛选不合适的客人，但不希望必须解释冻结理由。

这个设计不是技术问题，而是运营策略。我们把它写进了系统规则。

## 首页装修：从文字配置变成所见即所得

早期首页后台只是模块文字列表。用户看了以后说：能显示实际展示效果吗？不是文字，我想所见即所得。

于是首页装修逐渐变成三栏：

- 左边：添加模块。
- 中间：手机预览。
- 右边：模块设置。

用户又继续指出问题：

- 右侧设置排版重叠。
- 全屏时中间留白太多，挤压两侧。
- 右侧设置应该固定，不随中间滚动。
- 拖动模块到上下边缘时，页面应该跟着滚动。
- 左边模块可以一行三个，紧凑一点。

模块命名也被用户改得更清楚：

- 魔方入口 -> 魔方。
- 服务 -> 活动。
- 专题列表 -> 专题。
- 领队列表 -> 领队。
- 评价动态 -> 评价。

后来又增加：

- 图片广告。
- 标题文本。
- 辅助分割线。
- 文本框。
- 文字导航。
- 即将出发。
- 博客。

这一段体现了一个真实过程：一开始我们能做“功能”，但用户不断把它推向“可以日常使用的工具”。

## 首页风格：从泛旅行页调整成“大理自然体验”

用户给了多张参考图，整体偏自然、干净、安静。我们逐步调整客户首页：

- 品牌从 `DALITRIP` 改成“苍山徒步之家”。
- 风格不做大营销，不做花哨渐变。
- 字体朴素优雅。
- 颜色更接近米白、黑字、自然图片。
- 后来用户不喜欢某个大图 Hero，要求去掉。

首页里几个很重要的“有生命力”的模块被确认：

1. 即将出发的旅行  
   展示未来已经有人预订的活动，让客人看到有人正在参加，产生 follow 的感觉。

2. 最近的体验  
   展示最新评价，带头像、名字、日期、星级、图片。

3. 活动推荐  
   后来尝试借用博客卡片的文字、配色和布局。

4. 博客  
   放在首页最后，作为更像生活记录、旅行故事的内容。

用户多次指出细节：

- 图片不要贴边。
- 日期不要显示 2026，用“6 月 4 日”。
- 不要时钟图标。
- 即将出发图片放在文字上面。
- 一行放三张，可滑动。
- 图片圆角。

这些细节把首页从“能看”变成“有气质”。

## 活动详情页：按真实预约路径重排

活动详情页一开始不是用户想要的样子。用户给了参考图，要求：

- 进入活动页后按日期预约。
- 接下来显示最新一条评论。
- 点击查看全部进入活动评论列表。
- 再下面是详情。

我们重排为：

1. 活动头部。
2. 预约、评论、详情 tab。
3. 日期选择。
4. 时段选择。
5. 评论摘要。
6. 详情内容。
7. 喜欢这个的人也去了。

后来又修正：

- 点击活动时默认停留页首。
- 默认选第一个可订日期。
- 可约日期深色，不可约浅灰。
- 时段卡片不要太高。
- “已有 3 人报名，最多 12 人”改为“3 人预订，最多 12 人”。
- 时段卡片里显示最高规格价格。
- 详情里重复的时间、人数、价格去掉。
- 年龄、集合地点、领队微信做成同一格式。
- 导航按钮放集合地点后面。
- 领队微信加“复制”。
- 可能带队的领队挪到体验内容下面。

这部分最能体现：活动详情不是展示页，而是一个帮助客人做决定的页面。

## 评论：从简单列表到接近真实社区内容

评论系统经历了很多轮。

最早确认规则：

- 任何人任何时间都可以评价。
- 可以多次评价。
- 客人可以删除自己的评价。
- 管理员可以隐藏但不能删除。
- 评论者名字可以自己改。
- 字数 1500。
- 图片最开始说 16 张，后来发现手机上会挤，改回 9 张。
- 视频最多 1 个。
- 管理员、领队、其他用户可回复。

首页评论模块不断修：

- 不要普通列表，要横向滚动。
- 头像要保留，因为生动。
- 头像要和文字对齐。
- 正文和上面的正文对齐。
- 正文字体不要太大、不要太粗。
- 展开要在第三行末尾，不要另起一行。
- 展开后最多八行，更多内容在卡片内滑动。
- 图片左对齐。
- 图片不要太开。
- 多图布局参考用户截图。
- 图片有更多，进入图片集。
- 图片集背景先试浅蓝，后来觉得刺眼，改回黑色。
- 照片做圆角。

这段有很多“错了再改”的地方，尤其是展开位置、字体、图片布局。最后逐渐接近用户想要的真实评论氛围。

## 领队系统：从活动附属信息变成一个内容入口

用户提出需要一个领队信息库。

一开始只是活动里可能显示领队，后来扩展为：

- 领队有照片、名字、详细介绍。
- 领队介绍也需要多媒体编辑器。
- 活动可以勾选可能带队的领队。
- 客户点进领队档案，可以看到他可能带什么活动。
- 喜欢某个领队，就可以去预订相关活动。
- 还需要一个领队主页，展示全部领队，并有类似博客文章的总介绍。

这让系统不只是卖活动，也开始展示“人”。这对深度旅行很重要。

## 博客：从 Shopify/公众号内容迁移，到小程序内容系统

用户提出想建博客，可以评论，并给了参考图“我们的大理生活”。

我们确认博客需要：

- 标题。
- 封面图。
- 日期。
- 正文。
- 摘要。
- 如果摘要不填，取正文前 150 字。
- 显示/隐藏开关。
- 可评论。

用户喜欢博客模块的配色和文字，于是又要求把活动推荐也试试这种配色和布局。

博客后台一开始内容是 HTML 格式，用户说不行，需要正常编辑格式。后来我们加入更友好的编辑方式。

迁移方面，用户问公众号和 Shopify 都能不能迁移到博客。我们确认可以，但 Shopify 需要 API 权限；公众号如果没有导出能力，则可能要通过后台可见内容或其他方式处理。

期间也出现一个错误：我曾用临时图片补了没有封面的博客文章，用户指出不对，应该去 Shopify 找回原文封面。这是重要修正：迁移不是“补一个差不多”，而是尽量保留原文真实内容。

## 数据迁移：从天天预约、Shopify、Judge.me 来

数据迁移分几条线：

### 天天预约

用户登录了 Chrome 里的天天预约后台。我们尝试从现有后台迁移活动和组。

用户确认：

- 活动基本字段能看到。
- 小组也迁移一个。
- 后来“都导入了”“都对”。

过程中用户特别强调：

- 不要自动加 tag。
- 迁移来的活动先保持原样。
- 有些活动改名了，最好保留原来的活动名称。

### Shopify

用户希望从 Shopify 产品库导入厂商为 `Dali Experience Community` 的产品：

- 导入到活动。
- 包括图片和视频。
- 下载存到本地。

用户在 Shopify 后台创建应用时，我指导选择从 Dev Dashboard 创建应用，用于生成 Admin API 凭据。

### Judge.me

用户提到 Shopify 评论插件是 Judge.me。我们确认后续可以迁移评论，但暂时先不做。

## 后台活动列表：从能编辑到能日常管理

活动列表最早缺少编辑键，用户指出后，我们加了编辑。

后来又加：

- 删除活动。
- 暂停排班。
- 暂停排班和无排班的活动排到后面。

用户没看到删除键，要求截图确认。之后又要求删除旁边加编辑、暂停。

活动相册后台也是不断修出来的：

- 最初没有相册管理。
- 加入上传、删除、全览、拖曳排序。
- 用户觉得占空间，于是改成默认只显示一排。
- 点击最右侧“更多”，才进入完整编辑界面。
- 删除按钮要和上传照片放一起。
- 支持多选删除。
- 去掉前移/后移。
- 当前还存在一个待修问题：后面的图可以往前拖，前面的图不能往后拖。

## 过程中的错误和修正

这个项目不是一次做对的，很多地方是用户试用后指出问题，再修。

比较典型的修正包括：

- 排班一开始做得像一次性日期，后来改成长期星期规则。
- 特殊排班一开始有日期区间，后来改成只选一天并导入当天原排班。
- 活动详情一开始不是预约优先，后来改成日期预约、评论、详情的结构。
- 评论图片一开始太多太挤，从 16 张改回 9 张。
- 评论正文展开位置、字体、头像对齐多次不对，逐步修到接近参考。
- 首页装修一开始是文字列表，用户要求所见即所得。
- 右侧设置栏一开始会跟着滚动，后来改成固定。
- 后台相册一开始太占空间，改成一排预览 + 更多展开。
- 博客缺封面时，我曾用临时图片补，用户指出应从 Shopify 找真实封面。
- 有时本地服务挂了，页面出现 localhost 拒绝连接，需要重新启动。
- 系统偶尔出现 429 Too Many Requests，意思是请求过多，稍等后再试。

这些错误不是坏事，它们让系统越来越贴近真实使用。

## 这套小程序逐渐形成的样子

到目前为止，它已经不只是一个“旅游线路预订小程序”，而是一套围绕大理自然深度体验的内容和预约系统。

它包含：

- 活动。
- 排班。
- 订单。
- 微信支付。
- 自动退款规则。
- 顾客冻结与钱包。

## 这段沟通最值得保留的部分

这个小程序不是一开始就有完整蓝图，而是通过大量具体问题一点点长出来的：

- 看到旧系统哪里好用，就保留。
- 看到旧系统哪里不适合，就改成自己的。
- 原型做出来后，马上用真实眼光看哪里别扭。
- 每个小问题都修：按钮颜色、文字长度、图片圆角、日期格式、相册是否占空间。
- 业务规则不只写在后台，也体现在客户看到的每一句提示里。

最后形成的是一个很实际的方法：

先把业务讲清楚，再把页面试出来，然后让真实使用感不断修正产品。
